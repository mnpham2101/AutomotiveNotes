8.1 20-Day Training Plan

Day-by-Day Plan

Phase 1 — Process & Architecture Foundations

  • Day 1 — ASPICE
  • Read 1.1 ASPICE Overview in full.
  • Write out, from memory, the SWE.1–SWE.6 process list and one Tech Lead activity per process.
  • Answer all 1.1 Q&A questions without looking, then check against the file.
  • Day 2 — AUTOSAR Classic vs Adaptive
  • Read 2.1 AUTOSAR Architecture.
  • Redraw the layered architecture diagram from memory (App/RTE/BSW/MCAL).
  • Write a 3-sentence explanation of Classic vs Adaptive suitable for a customer call.
  • Day 3 — MCAL & Driver Layer
  • Read 2.2 AUTOSAR Classic Platform.
  • List 6 common MCAL modules and one peripheral each manages.
  • Explain, in writing, the difference between MCAL driver / ECU Abstraction / Complex Device Driver.
  • Day 4 — Review & Self-Test
  • Re-answer Day 1–3 Q&A sections cold.
  • Identify the 2 weakest topics so far and re-read just those sections.

Phase 2 — Platform & Embedded Fundamentals

  • Day 5 — Embedded Fundamentals
  • Read 3.1 Embedded Fundamentals.
  • Draw the Cortex-M boot chain (Boot ROM → stage-1 → stage-2 → app) from memory.
  • Explain Cortex-M vs Cortex-A (MPU vs MMU) in 2 sentences.
  • Day 6 — Linux Services & Daemons
  • Read 3.2 Linux Services & Daemons.
  • Write a systemd .service unit from scratch (no copy-paste) for a hypothetical daemon.
  • List the boot-to-ready chain for one of: Wi-Fi, Bluetooth, audio.
  • Day 7 — NXP Platform
  • Read 5.1 NXP Platform Overview.
  • List the NXP chip families relevant to this role and one use case each (S32K, S32G, i.MX).
  • Explain the difference between NXP's S32 Configuration Tool and a full BSW vendor tool (EB tresos).
  • Day 8 — Cross-Topic Integration Drill
  • Write a short narrative: "a CAN bus-off symptom is reported — trace it from application down to MCAL and possibly chip errata," citing which file covers each step.
  • Day 9 — Review & Self-Test
  • Re-answer Day 5–7 Q&A sections cold.
  • Mock-explain (out loud, timed 5 minutes) "MCAL vs driver vs bootloader" as if to a new hire.

Phase 3 — Hands-On Worked Example

  • Day 10 — Study the Worked Example
  • Read 4.1 MCAL Application Design end to end.
  • Annotate the architecture diagram with where each FR (FR-1..FR-5) is implemented.
  • Day 11 — Toolchain Setup
  • Confirm arm-none-eabi-gcc (or equivalent) is installed and runs (arm-none-eabi-gcc --version), or install it.
  • Read through the sample Makefile line by line and explain each flag's purpose in writing.
  • Day 12 — Build Attempt
  • Recreate a minimal version of the project structure (src/, startup/, linker/) and attempt to compile a trivial stand-in for one MCAL call (a stub function), even without real RTD sources.
  • Note every error encountered and how it was resolved — this log is valuable interview material.
  • Day 13 — Yocto Read-Through
  • Read 7.2 Yocto Build System.
  • Trace the build script's 6 stages and map each to the equivalent stage in the 4.1 Makefile.
  • Day 14 — Review & Self-Test
  • Re-answer Day 10–13 Q&A sections cold.
  • Write a 1-paragraph effort estimate for "add a second CAN channel" (per the 4.1 Q&A prompt) as if presenting to a customer.

Phase 4 — Lifecycle & Tooling Breadth

  • Day 15 — ECU Lifecycle
  • Read 6.1 ECU Development Lifecycle.
  • Redraw the V-model and label each phase with its ASPICE process ID.
  • Fill in the milestone-to-work-product table from memory, then check against the file.
  • Day 16 — Tech Stack Overview
  • Read 7.1 Tech Lead Tech Stack.
  • List every tool category and one specific tool per category without looking.
  • Day 17 — Tooling Decision Drill
  • Write a short justification (as if to management) for investing in better static analysis tooling, using ASPICE/risk framing from 7.1 Q&A.

Phase 5 — Leadership Practice & Synthesis

  • Day 18 — Communication Drills (JD Section 1)
  • Draft a mock customer email explaining a schedule risk caused by an NXP RTD version upgrade (combine 5.1 + 6.1 reasoning).
  • Practice a 2-minute spoken explanation of AUTOSAR Classic vs Adaptive, timed.
  • Day 19 — Team Management Drills (JD Section 2)
  • Draft a sample sprint/milestone plan for a 10-engineer team covering one V-model cycle (per 6.1).
  • Write 3 mentoring talking points for a junior engineer struggling with MISRA compliance.
  • Day 20 — Full Synthesis Review
  • Re-answer every Q&A section across all 11 files, cold, in one sitting (time-box: 90 minutes).
  • Identify remaining weak spots and schedule targeted re-reading.
  • Re-read 0.0 Introduction and confirm you can explain the whole knowledge base's structure unaided.

Phase 6 — Advanced Topics: Knowledge-Sharing Extension

Each day below names a suggested topic rather than a single file, because this knowledge base only summarizes it (or doesn't cover it at all) — that's the point. Per day: (1) read the cited source section first so the research has a baseline, (2) chase down at least one additional document/material on the topic (an official AUTOSAR spec, an NXP app note, a recognized blog/whitepaper) beyond what's cited here, (3) produce the listed knowledge-sharing output and, where possible, actually walk a colleague through it.

  • Day 21 — CP Software Clusters & Modular BSW Updates
  • Read AUTOSAR Layered Software Architecture (Doc ID 53), section on CP Software Clusters (Host Software Cluster vs Application Software Clusters, SwCluC, Binary Manifest).
  • Research goal: find one public OEM/Tier-1 source on why splitting monolithic BSW into independently buildable/updatable clusters matters for OTA software updates; summarize in 5 bullets.
  • Knowledge-sharing output: a 1-page explainer of Host vs Application Software Cluster for a colleague unfamiliar with the concept.
  • Day 22 — Multi-Core & Mixed-Criticality BSW Distribution
  • Read the same spec's sections on Content of Software Layers in Multi-Core Systems and ...in Mixed-Critical Systems (partitions per core, master/satellite EcuM, IOC, ASIL-vs-QM BSW module placement).
  • Research goal: pick one real multi-core automotive MCU (e.g. an NXP S32G/S32K3 family part) and sketch how BSW partitioning would map onto its cores.
  • Knowledge-sharing output: a diagram showing BSW module placement across 2 partitions/cores for a mixed-ASIL ECU, with the reasoning for each placement.
  • Day 23 — OS-Application Partitioning, Error Containment & Restart
  • Read the spec's Integration and Runtime Aspects → Partitioning section (Protection Hook, OsRestartTask, APPLICATION_ACCESSIBLE/RESTARTING/TERMINATED, TerminateApplication()).
  • Research goal: in the AUTOSAR OS specification, identify what actually triggers an application restart vs. a full termination.
  • Knowledge-sharing output: a short incident-style write-up — "SW-C X overruns its time budget — trace what happens from the Protection Hook through to RTE restart/cleanup."
  • Day 24 — BSW Scheduling, Data Consistency & Mode Management
  • Read the spec's BSW Scheduling (BSW Scheduler, BswTimingEvent/BswBackgroundEvent/etc., SchM_Enter/SchM_Exit) and Mode Management (BswM mode arbitration, Vehicle/Application/BSW mode interplay) sections.
  • Research goal: work out when a real integrator would choose interrupt-blocking critical sections vs. sequencing-only (no-op) SchM_Enter/SchM_Exit macros for two BSW modules sharing a resource.
  • Knowledge-sharing output: trace one mode request from a vehicle-level signal down through BswM to a concrete BSW mode switch, as a short annotated diagram.
  • Day 25 — Error Handling Taxonomy & Diagnostics (Det/Dem/Dlt)
  • Read the spec's Error Handling, Reporting and Diagnostic sections (error type vs. error class taxonomy: Development/Runtime/Transient/Production errors; Det_ReportError, Dem_SetEventStatus, Dlt_SendLogMessage).
  • Research goal: classify 5 realistic ECU faults (e.g. EEPROM write failure, CAN bus-off, an out-of-range API parameter, a transient bit-flip, a missing periodic CAN message) into the four error classes and pick the correct reporting API for each.
  • Knowledge-sharing output: a fault → error-class → reporting-module → production-reaction table.
  • Day 26 — Functional Safety (E2E) & Security (SecOC/MACsec)
  • Read the spec's Safety: End-to-End (E2E) Communication Protection section — note explicitly that the legacy E2E wrapper pattern is superseded; new projects should use the standardized E2E transformer — and the Security: SecOC / MACsec Key Agreement sections.
  • Research goal: compare E2E protection (signal-level CRC + counter), SecOC (MAC + freshness counter), and MACsec (Ethernet link-level ICV) — which threat (corruption vs. spoofing vs. replay) each one actually defends against.
  • Knowledge-sharing output: a short comparison table suitable for a customer-facing security-architecture discussion.
  • Day 27 — Energy Management & Global Time Synchronization
  • Read the spec's Energy Management (Partial Networking / Partial Network Cluster, ECU Degradation) and Global Time Synchronization (StbM, CanTSyn/FrTSyn/EthTSyn, Secure Global Time Sync) sections.
  • Research goal: design a Partial Network Cluster mapping across a small 3–4 ECU CAN example and walk through the wake/sleep flow; note which bus types (CAN, LIN, FlexRay, J1939) restrict or forbid Partial Networking and why.
  • Knowledge-sharing output: a redrawn PNC wake/sleep diagram plus 2–3 sentences on why sensor-fusion ECUs need synchronized global time.
  • Day 28 — AUTOSAR Methodology in Practice: the Official Workflow Example
  • Read the AUTOSAR Workflow Example supplementary material (Doc ID 1145) in full — the worked WheelSpeed/VehicleSpeed two-ECU CAN example, including its top-down vs. bottom-up System Description approaches (RootSwCompositionPrototype, DataMapping/SenderReceiverToSignalMapping attachment point) and its Appendix B meta-model class reference.
  • Research goal: locate the official WorkflowExample package from the AUTOSAR Classic Platform standards page and inspect the real WheelSpeed/VehicleSpeed System Extract ARXML — identify the SensorActuatorSwComponentType and EcuAbstractionSwComponentType elements in the actual XML.
  • Knowledge-sharing output: trace one signal end to end through the ARXML (sensor SW-C → CAN → consuming SW-C) citing the real XML element names found, and compare it against the 2.1 Sample call chain and the 6.1 4-step methodology.
