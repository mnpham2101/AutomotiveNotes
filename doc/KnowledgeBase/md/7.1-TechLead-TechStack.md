# 7.1 Tech Lead Tech Stack Overview

[← Home](0.0-Introduction.md)

## Concept Introduction

- Beyond AUTOSAR/MCAL domain knowledge, a Technical Lead in this role needs working fluency across a **toolchain spanning embedded build systems, version control, CI, static analysis, requirements/defect tracking, and (for mixed Cortex-A systems) Linux build tooling**.
- This file is an **index/overview**; the Yocto build system gets its own deep dive in [7.2](7.2-Yocto-Build-System.md) because of its size and direct relevance to the JD's "necessary knowledge" ask.

## Scope — Categorized Stack

- **Languages**: **C** (primary, MISRA-governed, MCAL/BSW/application code), **C++** (occasionally for tooling/test harnesses, rarely production MCAL code), **Python** (build scripts, test automation, internal tooling), **shell/Bash** (CI glue, Yocto/Linux side).
- **Embedded build & config tooling**: NXP **S32 Design Studio** + **S32 Configuration Tool** ([5.1](5.1-NXP-Platform-Overview.md)), **GNU Make**, **CMake** (increasingly used to wrap cross-compilation for portability across IDE/CI), **Elektrobit tresos** / **Vector DaVinci** (BSW config, where used).
- **Version control & review**: **Git** (branching strategy literacy — trunk-based vs. release-branch models common in automotive due to long-lived variant maintenance), Git hosting w/ merge-request review workflows (GitLab/Bitbucket/Azure DevOps are common in automotive more than GitHub).
- **CI/CD**: **Jenkins** (most common in automotive embedded due to on-prem/license-tool integration needs), increasingly **GitLab CI**; pipelines typically run: build → static analysis → unit test → HIL/target test trigger.
- **Static analysis & MISRA compliance**: **Polyspace** (MathWorks), **Helix QAC / Klocwork** (Perforce), **Coverity** — code review gate inputs (JD 3.2).
- **Requirements/defect/change management**: **IBM DOORS / DOORS NG**, **Polarion**, **JIRA** (often paired — DOORS/Polarion for formal requirements traceability, JIRA for day-to-day defect/task tracking) — underpin SUP.9/SUP.10 processes from [1.1](1.1-ASPICE-Overview.md).
- **Debug/trace**: **Lauterbach Trace32**, **P&E Multilink**, **S32 Debug Probe**, logic analyzers (Saleae) for bus-level (CAN/SPI) debugging.
- **Linux/Yocto side** (for Cortex-A companion systems): **Yocto/BitBake**, **devtool**, **systemd** ([3.2](3.2-Linux-Services-Daemons.md)), **U-Boot**, **buildroot** (lighter alternative to Yocto on some smaller projects), **Autotools** (`autoconf`/`automake`/`libtool` — still found wrapping many upstream open-source packages consumed inside Yocto recipes or built standalone for a Linux target).
- **Communication/protocol tooling**: CAN bus tools (**Vector CANoe/CANalyzer**, **PEAK PCAN-View**), diagnostic tooling (UDS/ISO 14229 testers).

## Use Cases

- **Choosing CMake vs. raw Makefiles for a new MCAL project**: CMake pays off when the same sources must build under multiple toolchains (S32DS-bundled GCC, a CI Docker image, a different customer's IAR toolchain) — a Tech Lead's tooling decision directly affects long-term maintenance cost, feeding JD 3.4 (technical decision-making).
- **Setting up Jenkins for a mixed MCAL + Yocto project**: typically two pipelines — a fast MCAL build+static-analysis pipeline (minutes) and a slower full Yocto image build pipeline (potentially hours), gated differently in the merge workflow.
- **MISRA gate tuning**: deciding which MISRA deviations are pre-approved project-wide vs. requiring per-instance justification in code review — a recurring code-review-policy decision for the Tech Lead (JD 3.2).
- **Mentoring**: junior engineers often need guided ramp-up on Git workflow and static-analysis-tool literacy before they can contribute productively to review-gated code — direct tie to JD 2.3.

## Sample — Representative Jenkins Pipeline Stage List

```groovy
// Jenkinsfile (declarative, simplified/illustrative)
pipeline {
    agent { label 'mcal-build-node' }
    stages {
        stage('Checkout')        { steps { checkout scm } }
        stage('Generate Config') { steps { sh './tools/gen_mcal_config.sh' } }
        stage('Build')           { steps { sh 'make -C build all' } }
        stage('Static Analysis') { steps { sh 'polyspace-bug-finder -results-dir reports/' } }
        stage('Unit Test')       { steps { sh 'ceedling test:all' } }
        stage('Archive')         { steps { archiveArtifacts 'build/*.elf, reports/**' } }
    }
}
```

## Q&A

- **Q: Why do automotive teams favor Jenkins over newer cloud-native CI (GitHub Actions, etc.)?**
  A: On-prem licensing for tools like Trace32/Polyspace/HIL rigs, plus long-lived programs (10+ years of variant maintenance) favor self-hosted, customizable CI infra that integrates with specialized lab hardware — though this is gradually shifting as more tooling moves to containerized/cloud forms.
- **Q: What's the role of Autotools in a Yocto-based project if Yocto has its own recipe format?**
  A: Many upstream open-source components Yocto packages (e.g. various GNU/Linux utilities) are themselves built with Autotools internally — a Yocto recipe for such a package often just wraps a standard `./configure && make && make install` sequence via the `autotools` bbclass. Understanding Autotools helps when patching or debugging those upstream recipes.
- **Q: How would you justify investing in better static analysis tooling to a customer or management?**
  A: Frame it in ASPICE/risk terms — static analysis catches MISRA/defect classes earlier (cheaper to fix at SWE.3/SWE.4 than after SYS.4 integration, see [6.1](6.1-ECU-Development-Lifecycle.md)), directly reducing field-defect risk a customer cares about contractually.
