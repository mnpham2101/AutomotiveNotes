# Interrupt Latency — Priorities, Recurrent Interrupts, and Real Cycle Costs

How long does a device wait between raising an interrupt request and its handler actually running — and how can that wait be *guaranteed* short enough? This document follows MIT 6.004 *Computation Structures*, "Devices & Interrupts" [1], continuing where [3.4.2 I/O with Interrupts](3.4.2-Embedded-Fundamentals-IOInterrupt.md) stopped: that document built the ISR/buffer/SVC machinery; this one asks how *quickly* the ISR side gets the CPU. It closes with measured entry/exit cycle counts on real Cortex-M cores — a cost the course's model treats as free — with the exception mechanics themselves covered in [3.4 Exceptions & Interrupts](3.4-Embedded-Fundamentals-Interrupt.md).

---

## 1. Concept Introduction

**Interrupt latency** is "the amount of time that elapses between a request to run some code and when that code actually starts executing" [1]. Latency matters because devices impose **deadlines**: a UART character must be read before the next one overwrites it; a disk sector must be consumed before the next sector arrives under the head. If a request must be fully serviced within deadline `D` and the handler's **service time** is `S`, then the worst-case latency the system may impose is:

```
L_max = D − S        (the handler must start by D − S to finish by D)
```

So latency analysis is really *deadline* analysis: for each device, is the worst-case latency the rest of the system can inflict smaller than `L_max`?

### Why first-come-first-served is not enough — the need for priorities

A running example with three devices [1] shows the problem:

| Device   | Service time `S` |
| -------- | ---------------- |
| Keyboard | 800 µs           |
| Disk     | 500 µs           |
| Printer  | 400 µs           |

With simple **first-come-first-served** dispatch (whoever asked first runs, and runs to completion), each device's worst case is: every *other* handler got in line just before it. Worst-case latencies come out as keyboard 900 µs, disk 1200 µs, printer 1300 µs — i.e. "long-running handlers have a huge impact on the worst-case latency seen by the other devices" [1]. The slow keyboard handler is allowed to sit in front of the disk, whose deadline may be far tighter. FCFS has no way to encode *urgency*; that is what **interrupt priorities** add. A natural assignment strategy is **earliest deadline**: "sort the requests by their deadlines. Assign the highest priority to the earliest deadline, second priority to the next deadline, and so on" [1] — the static-priority cousin of EDF scheduling.

Priorities come in two strengths, detailed in sections 2 and 3:

- **Non-preemptive ("weak") priorities** — priority decides which *pending* request is dispatched next, but "the current task always runs to completion even if a higher-priority request arrives while it's executing" [1]. Cheap to implement (a sorter in front of the dispatcher), but every device can still be blocked once by whatever lower-priority handler happens to be running.
- **Preemptive ("strong") priorities** — a higher-priority request *suspends* the currently running lower-priority handler, nesting interrupt contexts. High-priority latency becomes independent of everyone slower; the price is hardware support for a current-priority level and re-entrant kernel code.

Finally, real devices do not interrupt once — they interrupt **recurrently**, each with its own period. That changes the question from "who waits for whom once?" to "does the sum of all recurring service demands fit inside every device's deadline window, forever?" — the interrupt-load and window analysis of section 4, and the reason this topic is the foundation of rate-monotonic thinking in RTOS-based ECU software.

## 2. Non-Preemptive (Weak) Priorities

Under weak priorities the dispatcher always picks the highest-priority *pending* request, but never interrupts a running handler. The worst case for device `X` is therefore [1]:

1. an instant ago, the *longest* lower-priority handler started running — it cannot be preempted, so `X` eats its full service time once, plus
2. the service time of every *higher*-priority device, since pending higher-priority requests are dispatched before `X`.

With the earliest-deadline ordering disk > printer > keyboard:

| Device   | Blocked by (lower, one worst case) | Higher-priority work    | Worst-case latency |
| -------- | ---------------------------------- | ----------------------- | ------------------ |
| Disk     | keyboard (800 µs)                  | —                       | **800 µs**         |
| Printer  | keyboard (800 µs)                  | disk (500 µs)           | **1300 µs**        |
| Keyboard | —                                  | disk + printer (900 µs) | **900 µs**         |

Compared with FCFS, the disk improved from 1200 µs to 800 µs — but it is still hostage to the keyboard's 800 µs body, because nothing can interrupt a handler that has already started. That residual term — *max service time of any lower-priority handler* — is exactly what strong priorities remove.

> Automotive echo: a "weak-priority" situation reappears any time an ISR (or a critical section with interrupts locked) runs to completion regardless of what arrives — which is why coding rules keep ISR bodies and interrupt-locked regions short: their length is a *latency tax on every higher-priority event in the system*.

## 3. Preemptive (Strong) Priorities

Strong priorities let a higher-priority request interrupt a running lower-priority handler: the hardware tracks the priority level currently executing and accepts a request only if its priority is strictly higher [1]. On Cortex-M this comparison is performed by the NVIC (see the Q&A).

Same devices, same ordering (disk > printer > keyboard):

| Device   | Can be delayed by         | Worst-case latency     |
| -------- | ------------------------- | ---------------------- |
| Disk     | nothing — preempts anyone | **0 µs**               |
| Printer  | disk only                 | **500 µs**             |
| Keyboard | disk + printer            | **900 µs** (unchanged) |

The headline property: "a strong priority system allows low-priority handlers to be interrupted by higher-priority requests, so the worst-case latencies seen at high priorities is unaffected by the service times of lower-priority handlers" [1]. The lowest-priority device gains nothing — it was already waiting for everyone — but the tightest deadlines become schedulable *by design* rather than by hoping slow handlers finish in time. The costs are re-entrancy (each preemption stacks another interrupt context) and kernel data structures that tolerate being interrupted mid-update.

## 5. What Entry and Exit Actually Cost — Cycle Counts on Real Cores

The scheduling analysis in the preceding sections treats service time `S` as given and dispatch as free, noting only that "it takes time to save the process state, switch to the kernel context, and dispatch to the correct interrupt handler" and that designers should minimize that setup code [1]. On real silicon this overhead is documented, and it belongs *inside* `S` (and the entry part inside the latency `L`) in any latency budget.

For ARM Cortex-M — the class of core in NXP S32K MCUs — interrupt entry is done in hardware: the core finishes/abandons the current instruction, pushes R0–R3, R12, LR, PC and xPSR (8 words) onto the stack, fetches the vector, and refills the pipeline. With **zero-wait-state memory**, ARM specifies [2][3][4]:

| Core       | Entry (request → first ISR instruction) | Exit (return to interrupted code)                          |
| ---------- | --------------------------------------- | ---------------------------------------------------------- |
| Cortex-M0  | 16 cycles                               | — (not specified in [2])                                   |
| Cortex-M0+ | 15 cycles                               | — (not specified in [2])                                   |
| Cortex-M3  | 12 cycles                               | 12 cycles [4]                                              |
| Cortex-M4  | 12 cycles (max) [3]                     | 10 cycles (return-target fetch overlaps the stack pop) [3] |

So a complete *raise → handle → return to main* round trip on a Cortex-M3/M4 costs roughly **22–24 cycles of pure hardware overhead** plus the handler body — e.g. ~0.15 µs at 160 MHz — and because the stacking already saves the caller-saved registers, the ISR can be a plain C function with no assembly-language wrapper [2].

Two hardware optimizations shave the overhead further when interrupts are back-to-back [2][5]:

- **Tail-chaining**: if another interrupt is pending when a handler returns, the unstack/restack pair is skipped and the next handler is entered in **6 cycles** (versus 12 + 12), "saving 18 cycles in total" [5].
- **Late arrival**: if a higher-priority interrupt arrives *during* the stacking for a lower-priority one, the (priority-independent) stacking is reused and the core simply fetches the higher-priority vector instead.

And several effects stretch the numbers in practice, which is why the zero-wait-state figure is a floor, not a promise [2][3]:

- **Flash wait states** — at automotive clock rates, vector and stack accesses rarely hit zero-wait-state memory; each wait state inflates entry/exit.
- **FPU context (Cortex-M4F/M7)** — a floating-point-active exception frame is 26 words, not 8; *lazy stacking* (the default) defers the FP save so entry stays at 12 cycles unless the ISR itself uses the FPU.
- **Priority masking / higher-priority activity** — a masked or lower-priority request by definition waits; this is exactly the `L` analyzed in sections 2 and 3, stacked *on top of* the hardware entry cost.

The division of labor: architecture manuals bound the *mechanical* cost (≈12–16 cycles each way); the priority analysis of sections 2 and 3 bounds the *scheduling* cost. A realistic latency budget adds both.

## 6. Q&A

1. **Why doesn't a strong-priority system improve the lowest-priority device's latency at all?**

   A: The lowest-priority device was never blocked by anything *below* it — its worst case was always "all higher-priority work first" (keyboard: 500 + 400 = 900 µs in both tables). Preemption only removes the *lower-priority-handler-already-running* term, and the bottom device never had one. Strong priorities protect the top of the priority stack, not the bottom.

2. **Is the Cortex-M NVIC a weak or strong priority system?**

   A: Strong — a pending exception whose (group) priority is higher than the running one preempts it immediately; the comparison uses the NVIC priority registers, further gated by the `BASEPRI`/`PRIMASK` masks. Within one *group* priority, Cortex-M *sub-priority* behaves weakly: it only orders pending exceptions and never preempts — so one NVIC configuration actually contains both schemes from sections 2 and 3.

3. **Where do the 12/16 entry cycles go, and why does the C ISR need no assembly wrapper?**

   A: The hardware spends them pushing the 8-word exception frame (R0–R3, R12, LR, PC, xPSR), fetching the vector, and refilling the pipeline. Since the frame is precisely the caller-saved register set of the AAPCS calling convention, the handler can be an ordinary C function: anything else it touches, the compiler already saves/restores as callee-saved [2].

4. **Which number should go into a latency budget as a handler's service time `S` on a Cortex-M4?**

   A: `S` = entry (12 cycles) + handler body (measured worst case, including flash wait states and any FP stacking the body triggers) + exit (10 cycles) — with exit sometimes replaced by a 6-cycle tail-chain, but budget the larger value. Using only the body time undercounts every device's demand, and the error multiplies with recurrence: a 22-cycle overhead on a 1 ms-period interrupt is 22 000 cycles of hidden load per second.
