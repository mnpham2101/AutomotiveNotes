# Faults on the Arm Cortex-M7

A practical reference to the fault sub-category of the Cortex-M7 exception model:
what makes a fault different in policy from an ordinary interrupt, the four fault
handlers, the registers that report and control them, and the escalation/lockup
rules unique to faults. This document assumes the shared exception mechanics
(entry/return, stacking, the vector table, exception state) already covered in
[3.4 Exceptions & Interrupts](3.4-Embedded-Fundamentals-Interrupt.md) and does
not repeat them.

---

## 1. Concept Introduction

### What is a fault?

A **fault** is the sub-category of exception raised when the instruction stream
itself does something the core cannot allow — a bad memory access, an undefined
instruction, a divide-by-zero, a bus error [1][2]. Unlike an interrupt, nothing
external triggered it: the fault *is* the instruction currently executing (or,
for imprecise bus errors, one that already retired) [1].

### Fault vs interrupt — operational differences

Both share the same entry/return/stacking machinery (see 3.4), but differ in
policy [1]:

| Policy area          | Interrupt (IRQn)                                  | Fault                                                                          |
| -------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------ |
| **Enable**            | Per-IRQ bit in `NVIC_ISER`/`ICER`                  | Per-*type* bit in `SHCSR` (`MEMFAULTENA`, `BUSFAULTENA`, `USGFAULTENA`)         |
| **Software trigger**  | Yes — `NVIC_ISPR` or `STIR`                        | No direct equivalent; only the pending bits in `SHCSR` can be forced           |
| **Escalation**        | None — stays pending until serviceable              | Escalates to **HardFault** if it can't be serviced (see Section 4)             |
| **Lockup on re-fault**| Never                                              | Yes, if the fault recurs while already inside the NMI or HardFault handler     |
| **Priority**          | Configurable                                       | Configurable (MemManage/BusFault/UsageFault) or fixed −1 (HardFault)           |

### The four fault handlers

| Fault          | Priority     | Enabled at reset? | Typical cause                                    |
| -------------- | ------------ | ------------------ | ------------------------------------------------- |
| **HardFault**  | −1 (fixed)   | Always on          | Catch-all; escalation target for the other three  |
| **MemManage**  | Configurable | **No**             | MPU access violation                               |
| **BusFault**   | Configurable | **No**             | Bus/ECC error on an instruction or data access      |
| **UsageFault** | Configurable | **No**             | Undefined instruction, divide-by-zero, bad state    |

MemManage, BusFault, and UsageFault must be explicitly enabled via their
`SHCSR` bit; until then, any occurrence escalates straight to HardFault — see
the bring-up note in [3.4 §2](3.4-Embedded-Fundamentals-Interrupt.md#2-interrupt--exception-list-on-cortex-m7)
[1].

## 2. Fault Types & Causes

Each configurable fault reports *why* it fired through a dedicated status-bit
field, all packed into the **Configurable Fault Status Register (`CFSR`)** [1]:

| Cause                                             | Handler      | Status bit    | Sub-register |
| -------------------------------------------------- | ------------ | ------------- | ------------- |
| Bus error reading the vector table                 | HardFault    | `VECTTBL`     | `HFSR`        |
| Fault escalated because it couldn't be handled      | HardFault    | `FORCED`      | `HFSR`        |
| Instruction fetch from an XN region / MPU violation | MemManage    | `IACCVIOL`    | `MMFSR`       |
| Data access MPU violation                           | MemManage    | `DACCVIOL`    | `MMFSR`       |
| MemManage fault during exception stacking           | MemManage    | `MSTKERR`     | `MMFSR`       |
| MemManage fault during exception unstacking         | MemManage    | `MUNSTKERR`   | `MMFSR`       |
| MemManage fault during FPU lazy state preservation  | MemManage    | `MLSPERR`     | `MMFSR`       |
| Bus error during instruction prefetch               | BusFault     | `IBUSERR`     | `BFSR`        |
| Precise data bus error                              | BusFault     | `PRECISERR`   | `BFSR`        |
| Imprecise data bus error                            | BusFault     | `IMPRECISERR` | `BFSR`        |
| Bus error during exception stacking                 | BusFault     | `STKERR`      | `BFSR`        |
| Bus error during exception unstacking                | BusFault     | `UNSTKERR`    | `BFSR`        |
| Bus error during FPU lazy state preservation         | BusFault     | `LSPERR`      | `BFSR`        |
| Attempted coprocessor access                        | UsageFault   | `NOCP`        | `UFSR`        |
| Undefined instruction                               | UsageFault   | `UNDEFINSTR`  | `UFSR`        |
| Invalid instruction-set state                       | UsageFault   | `INVSTATE`    | `UFSR`        |
| Invalid `EXC_RETURN` value on return                | UsageFault   | `INVPC`       | `UFSR`        |
| Unaligned load/store (if trapping enabled)          | UsageFault   | `UNALIGNED`   | `UFSR`        |
| Divide-by-zero (if trapping enabled)                | UsageFault   | `DIVBYZERO`   | `UFSR`        |

### Synchronous vs asynchronous BusFault — precise vs imprecise

All bus faults triggered by **load** operations are synchronous (precise); all
faults triggered by **store** operations are asynchronous (imprecise) —
including stores to Device or Strongly-Ordered memory — because the write
buffer may retire the store before the error surfaces [1]:

- **Precise (`PRECISERR`)** — the stacked `PC` points at the instruction that
  caused the fault, and `BFAR` holds a valid address.
- **Imprecise (`IMPRECISERR`)** — the stacked `PC` has no relation to the
  faulting instruction, and `BFAR` is not written. If an IRQ fires right after
  the store, the write buffer may not have drained yet, so the fault can
  surface across a context boundary, in a handler that never issued the store.
- If the BusFault handler is disabled when an asynchronous fault occurs,
  HardFault is pended instead — and that particular HardFault **never**
  escalates into lockup, unlike a synchronous fault recurring inside NMI/HardFault
  [1].
- Debugger-initiated loads/stores are always synchronous, and visible only to
  the debug interface, not to application code [1].

## 3. Registers

### Fault status registers

`CFSR` is one word made of three byte/halfword sub-registers, each independently
byte-accessible [1]:

| Sub-register | Bits in `CFSR` | Covers        |
| ------------- | -------------- | ------------- |
| `MMFSR`       | `[7:0]`        | MemManage     |
| `BFSR`        | `[15:8]`       | BusFault      |
| `UFSR`        | `[31:16]`      | UsageFault    |

`HFSR` is a separate register for HardFault (`VECTTBL`, `FORCED`, and a
`DEBUGEVT` bit reserved for debug use).

All fault-status bits are **sticky**: once set, they stay set until software
writes `1` to clear them (write-one-to-clear), or until reset [1].

### Fault address registers

| Register | Set by                                   | Valid-flag                  |
| -------- | ----------------------------------------- | ----------------------------- |
| `MMFAR`  | A MemManage fault with a known address     | `MMFSR.MMARVALID`             |
| `BFAR`   | A *synchronous* BusFault with a known address | `BFSR.BFARVALID`          |

Because a later exception can overwrite these before the original handler reads
them, a fault handler must read the address register **before** checking its
valid flag, and re-check the flag every time — a higher-priority fault handler
that ran in between may have already clobbered it [1].

### Enabling and pending faults

`SHCSR` (System Handler Control and State Register) holds, per fault [1]:

- **Enable** — `MEMFAULTENA`, `BUSFAULTENA`, `USGFAULTENA` (HardFault has none;
  it can't be disabled).
- **Pending** — `MEMFAULTPENDED`, `BUSFAULTPENDED`, `USGFAULTPENDED`,
  `SVCALLPENDED` — software can set these directly, which is how an RTOS can
  force a fault-priority context switch.
- **Active** — `MEMFAULTACT`, `BUSFAULTACT`, `USGFAULTACT`, `SVCALLACT` — reads
  as 1 while that handler is running; writable, but only safe to change if the
  stacked context is adjusted to match, otherwise the next return corrupts state.

These registers sit in the same System Control Space as the NVIC registers —
see the memory map in
[3.4 §3](3.4-Embedded-Fundamentals-Interrupt.md#where-these-registers-live-in-the-mcu-memory-map).

## 4. Fault Escalation & Lockup

### Escalation rules

A fault with configurable priority is **escalated to HardFault** in any of these
cases [1]:

1. A fault handler causes the *same kind* of fault it is currently servicing
   (it can't preempt itself — same priority as what's running).
2. A fault handler causes a fault with the *same or lower* priority than the one
   it's servicing.
3. Any exception handler causes a fault with priority *same or lower* than the
   currently executing exception.
4. The fault occurs but its handler is **not enabled**.

> **Exception to the rule:** a BusFault occurring while *stacking for entry
> into the BusFault handler itself* does **not** escalate — the handler still
> runs, just with a corrupted stack; software must recognize this from
> `BFSR.STKERR` [1].

Only Reset and NMI can preempt HardFault; HardFault itself can preempt any
exception except Reset, NMI, or another HardFault [1].

### Lockup

The processor enters **lockup** if a fault occurs *while already executing* the
NMI or HardFault handler [1]:

- No instructions execute while locked up.
- Recovery is only possible via: a reset, an NMI, or a debugger halt.
- **Special case:** if lockup was entered from the NMI handler, a *subsequent*
  NMI does **not** bring the processor out of lockup.

Lockup is the hardware's last resort when a fault can no longer be safely
escalated (HardFault is already the escalation target and is currently
running) — the automotive-relevant takeaway is that a HardFault handler must
be written defensively (no further faulting operations) precisely to avoid
triggering this state.

## 5. Q&A

1. **Why can't MemManage/BusFault/UsageFault be left disabled in production code?**

   A: Any occurrence while disabled escalates straight to HardFault, which
   collapses three distinct, diagnosable fault types into one generic handler —
   you lose the specific cause (`CFSR` bit) that would otherwise tell you
   exactly what went wrong.

2. **If `BFAR` holds an address, is it safe to assume it caused the fault I'm currently handling?**

   A: Only if `BFSR.BFARVALID` is 1 *and* you read it before any other fault
   could have run and overwritten it. A higher-priority fault handler running
   in between can silently invalidate or overwrite `BFAR`/`MMFAR`.

3. **Does an imprecise BusFault mean the fault is less serious?**

   A: No — it means the *diagnostic information* is weaker (no reliable `PC`
   or `BFAR`), not that the underlying bus error is less severe. It's simply a
   side effect of store operations being asynchronous, so the CPU has already
   moved past the faulting instruction by the time the error surfaces.

4. **Can a fault ever be pending without being active, like an ordinary interrupt?**

   A: Yes — the same inactive/pending/active/active-and-pending model from
   [3.4 §1](3.4-Embedded-Fundamentals-Interrupt.md#exception-state) applies to
   faults too. The difference is only in what happens when a fault *can't* be
   serviced: it escalates, whereas an ordinary interrupt just stays pending.
