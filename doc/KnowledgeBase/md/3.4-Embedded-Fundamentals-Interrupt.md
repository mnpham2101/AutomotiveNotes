# Exceptions & Interrupts on the Arm Cortex-M7

A practical reference to how the Cortex-M7 core handles exceptions and interrupts:
the concepts, the full interrupt list, and the registers that control them.

---

## 1. Concept Introduction

### What is an exception?

An **exception** is any condition that changes the normal, sequential flow of a
program. An **exception handler** is the code that runs in response to it — also
called an Interrupt Service Routine (ISR) [1][2]. When an exception is taken, the
processor saves its current context to the stack and branches to that handler;
when the handler finishes, the processor restores the saved context and resumes
exactly where it left off [1].

On the Cortex-M7, the processor core and the **NVIC** (Nested Vectored Interrupt
Controller) together prioritize and manage every exception [1].

### What is an interrupt?

An **interrupt** is a *specific kind* of exception — one raised **asynchronously**
by hardware, typically a peripheral (a CAN frame arrives, a timer expires, a pin
changes state). The terms "interrupt" and "exception" are often used
interchangeably, but the precise relationship is [1]:

> **All interrupts are exceptions, but not all exceptions are interrupts.**

Arm defines an interrupt (IRQ) as an exception signaled by a peripheral or
generated by a software request, used by peripherals to communicate with the
processor — and **all interrupts are asynchronous to instruction execution**
(DUI0646 §2.3.2) [1].

Faults (illegal memory access, divide-by-zero) are exceptions too, yet they are
not "interrupts" because they are produced by the instruction stream itself rather
than by an outside event [1].

### Purpose — why the mechanism exists

- **Responsiveness without polling.** The CPU can run normal code and be notified
  the instant an event needs attention, instead of wasting cycles constantly
  checking status flags [1].
- **Real-time behavior.** Urgent events preempt less-urgent work through a strict
  priority scheme, giving deterministic latency [1].
- **Error handling.** Faults trap dangerous conditions (bad pointers, bus/ECC
  errors) so the system can recover or fail safely — essential for functional-safety-rated
  (ASIL) automotive MCUs.
- **System services & multitasking.** Special exceptions implement supervisor
  calls (`SVC`) and RTOS context switching (`PendSV`, `SysTick`) [1].

### Grouping the exception types

Cortex-M exceptions fall into a few natural families [1][2]:

| Group                          | Members                                                                    | Character                                              |
| ------------------------------ | -------------------------------------------------------------------------- | ------------------------------------------------------ |
| **Reset**                      | Reset                                                                      | Brings the core up from a known state                  |
| **Non-maskable**               | NMI                                                                        | Highest-priority event that software can never disable |
| **Faults**                     | HardFault, MemManage, BusFault, UsageFault                                 | Error/exception conditions from executing code         |
| **System service**             | SVCall, PendSV, SysTick, DebugMonitor                                      | OS/kernel support exceptions                           |
| **External interrupts (IRQn)** | Peripheral interrupts (CAN controller, UART, timer/PWM, DMA, GPIO pins, …) | Asynchronous hardware events via the NVIC              |

### Activation type — synchronous vs asynchronous

Every exception is either **synchronous** or **asynchronous**, and the difference
matters a lot when debugging [1][2]:

- **Synchronous** — caused *directly by the instruction currently executing*,
  and always reported **with** that instruction: the stacked return address
  points at the offending instruction, so you know exactly what triggered it
  (DDI0403E §B1.5.1) [1]. A bus/MPU fault on a load, an `SVC` call, an
  undefined instruction, and a divide-by-zero are all synchronous.
- **Asynchronous** — caused by an event **independent of the current
  instruction**: a peripheral, the NMI pin, or a buffered store that had
  already retired by the time the error surfaced. Reset, NMI, all external
  IRQs, PendSV, SysTick, and the imprecise BusFault are asynchronous. For any
  asynchronous exception other than reset, the processor can execute another
  instruction between when the exception is triggered and when it enters the
  handler (DUI0646 §2.3.2) [1].

#### Synchronous vs precise — related, not synonyms

The two word pairs are close but not interchangeable [1]:

- **Synchronous / asynchronous** classifies *every* exception by whether it is
  tied to the instruction currently executing, and drives the return-address
  rule: a synchronous exception stacks the address **of** the faulting
  instruction, an asynchronous one stacks the **next instruction not yet
  executed** (`ReturnAddress()`, DDI0403E §B1.5) [1].
- **Precise / imprecise** describes only how a *fault* is **reported**. A
  precise data-access fault is reported with its instruction and latches the
  faulting address into `BFAR`/`MMFAR`; an imprecise fault is reported with no
  fixed relationship to its instruction and leaves `BFAR` not updated
  (`BFARVALID` clear) [1].

For **faults** the two line up exactly — DUI0646 Table 2-14 gives the
BusFault's activation as synchronous when precise, asynchronous when imprecise,
and DDI0403E §A3.8.4 calls an imprecise fault an asynchronous exception. So for
a fault: **synchronous ⇒ precise, asynchronous ⇒ imprecise** [1].

The equivalence does **not** extend to interrupts: IRQs, NMI, PendSV, and
SysTick are asynchronous yet never called "imprecise" — they are taken at a
clean instruction boundary, stack a well-defined return address, and lose no
state. Precise/imprecise is fault-*reporting* vocabulary only [1].

> **Important special case — BusFault.** A bus fault from a **load** is synchronous
> (precise), but one from a **store** is asynchronous (imprecise), because the
> Cortex-M7 write buffer may retire the store before the error surfaces. This is
> exactly why a write-caused fault can arrive with the fault-address register
> (`BFAR`) marked invalid [1][2].

### Exception state

At any moment, every exception is in one of four states, tracked per-exception by
the NVIC/SCB pending and active bits [1][3]:

| State                  | Meaning                                                                                                                                                                               |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Inactive**           | Not pending, not active — nothing outstanding for this exception.                                                                                                                     |
| **Pending**            | The trigger condition has occurred, but the handler hasn't started yet.                                                                                                               |
| **Active**             | The handler has started running and hasn't returned yet.                                                                                                                              |
| **Active and pending** | The handler is running, *and* the same exception has fired again in the meantime — common with a peripheral that reasserts its interrupt line before the first one is fully serviced. |

Software can read and, for most exceptions, directly set these states through
the registers introduced in [3. Registers](#3-registers) below — `NVIC_ISPR`/
`ICPR` for pending, `NVIC_IABR` for active, with `ICSR` and `SHCSR` doing the
equivalent job for the system exceptions.

### Handler mode vs Thread mode

Cortex-M7 has exactly two processor modes [1]:

- **Thread mode** — runs normal application code. Entered on reset. Can execute
  as either privileged or unprivileged software, selected by `CONTROL.nPRIV`.
- **Handler mode** — runs exception handlers. Entered whenever an exception is
  taken. Always privileged, and **always uses the MSP** — the `CONTROL.SPSEL`
  bit that selects the PSP in Thread mode is ignored while in Handler mode.

### MSP vs PSP, and how the stack pointer moves

R13 (`SP`) is banked into two independent pointers, only one active at a
time, each pointing into its own stack region in RAM [1]:

- **Main stack**, pointed to by **MSP** — the reset default; the only stack
  Handler mode ever uses.
- **Process stack**, pointed to by **PSP** — usable only in Thread mode,
  selected by setting `CONTROL.SPSEL` to 1. Typical RTOS split: each task uses
  its own process stack while the kernel and all exception handlers use the
  main stack, so a task overflow can't corrupt the kernel's or another task's
  stack.

Both are ordinary **full-descending stacks**: whichever pointer is active
holds the address of the *last* item pushed. A push decrements the pointer
first, then writes; a pop reads first, then increments the pointer back up
[1]. Exception entry uses exactly this mechanism — see
[Exception stack frame](#exception-stack-frame) below for the eight-word frame
it pushes onto whichever stack (main or process) was active before the
exception.

## 2. Interrupt & Exception List on Cortex-M7

Lower priority number = higher urgency. Reset, NMI, and HardFault have **fixed**
negative priorities; everything else is **configurable** (0–255, subject to the
number of priority bits the vendor implements) [1][2].

| Exception                       | Priority     | Registers that configure / control it                                                                        | Activation                         | Purpose                                                                                            |
| ------------------------------- | ------------ | ------------------------------------------------------------------------------------------------------------ | ---------------------------------- | -------------------------------------------------------------------------------------------------- |
| **Reset**                       | −3 (fixed)   | `VTOR`, `AIRCR` (`SYSRESETREQ`)                                                                              | Async                              | Entry point after power-on / reset                                                                 |
| **NMI**                         | −2 (fixed)   | `ICSR` (`NMIPENDSET`)                                                                                        | Async                              | Critical, unmaskable events — typically an external NMI pin or a safety-monitor fault reaction [1] |
| **HardFault**                   | −1 (fixed)   | `HFSR`, `SHCSR`                                                                                              | Sync or Async (follows the escalated fault) | Catch-all fault; escalation target when a configurable fault is disabled                           |
| **MemManage**                   | Configurable | `SHPR1`, `SHCSR` (`MEMFAULTENA`), `CFSR→MMFSR`, `MMFAR`                                                      | Sync                               | MPU access violations                                                                              |
| **BusFault**                    | Configurable | `SHPR1`, `SHCSR` (`BUSFAULTENA`), `CFSR→BFSR`, `BFAR`                                                        | Sync (precise) / Async (imprecise) | Bus errors, incl. ECC faults on uninitialized RAM                                                  |
| **UsageFault**                  | Configurable | `SHPR1`, `SHCSR` (`USGFAULTENA`), `CFSR→UFSR`, `CCR`                                                         | Sync                               | Undefined instruction, divide-by-zero, unaligned access                                            |
| **SVCall**                      | Configurable | `SHPR2`                                                                                                      | Sync                               | Supervisor call — RTOS/kernel system-call entry                                                    |
| **DebugMonitor**                | Configurable | `SHPR3`, `DEMCR` (`MON_EN`)                                                                                  | Sync                               | Software (non-halting) debug events                                                                |
| **PendSV**                      | Configurable | `SHPR3`, `ICSR` (`PENDSVSET`/`PENDSVCLR`)                                                                    | Async                              | Deferred system work — classic RTOS context switch                                                 |
| **SysTick**                     | Configurable | `SHPR3`, `ICSR` (`PENDSTSET`), `SYST_CSR`/`RVR`/`CVR`                                                        | Async                              | RTOS tick / periodic timer                                                                         |
| **External IRQn** (peripherals) | Configurable | `NVIC_ISER`/`ICER` (enable), `ISPR`/`ICPR` (pending), `IABR` (active), `IPR` (priority), `STIR` (SW trigger) | Async                              | Peripheral events: CAN controller, UART, timer/PWM, DMA, GPIO pins, etc.                           |

**Masking is not per-exception.** Three core special registers gate whole groups
at once (they are *not* memory-mapped — reached only via `MRS`/`MSR`) [1]:

| Register    | Effect                                                              |
| ----------- | ------------------------------------------------------------------- |
| `PRIMASK`   | Blocks all configurable-priority exceptions; leaves NMI + HardFault |
| `FAULTMASK` | Blocks everything except NMI                                        |
| `BASEPRI`   | Blocks configurable-priority exceptions at or below a chosen level  |

> **Bring-up note:** MemManage, BusFault, and UsageFault are **disabled out of
> reset**. Until their `…FAULTENA` bits in `SHCSR` are set, those faults escalate
> straight to **HardFault** [1] — a common early bring-up symptom on any Cortex-M7
> target when these enables haven't been configured yet.

## 3. Registers

### What a register is

A **register** is a small, named hardware storage location — a handful of bits
with a fixed meaning — that software reads or writes to **control** hardware
behavior or to **observe** hardware state. The interrupt/exception registers are
**memory-mapped**: they sit at fixed addresses inside a reserved region of the
address map called the **System Control Space (SCS)**, so ordinary load/store
instructions (or CMSIS helpers) reach them [1].

### The role registers play for interrupts/exceptions

They cover four jobs [1]:

- **Enable / disable** — decide whether an exception is even allowed to fire.
  (`NVIC_ISER`/`ICER` for peripheral IRQs; `SHCSR` for the configurable faults.)
- **Pend / clear** — mark an exception as waiting to be serviced, or trigger one
  from software. (`NVIC_ISPR`/`ICPR`, `STIR`, and `ICSR` for NMI/PendSV/SysTick.)
- **Prioritize** — set how urgent each exception is and how preemption nests.
  (`NVIC_IPR` per IRQ; `SHPR1–3` for system exceptions; `AIRCR.PRIGROUP` splits
  priority into preempt vs sub-priority.)
- **Report / diagnose** — after a fault, record *what* went wrong and *where*.
  (`CFSR` = `MMFSR`+`BFSR`+`UFSR`, plus `HFSR`, `MMFAR`, `BFAR`.)

The masking special registers (`PRIMASK`, `FAULTMASK`, `BASEPRI`) add a fifth,
coarser job: temporarily silencing whole priority bands [1].

### Where these registers live in the MCU memory map

All of the registers above occupy one small, fixed corner of the Cortex-M7 address
space — the System Control Space at `0xE000_E000`, inside the Private Peripheral
Bus [1]. The diagram below zooms in from the full 4 GB map down to the individual
registers:

## 4. Exception Handler

An **exception handler** is the function that runs in response to a specific
exception — the code that implements whatever that exception requires (service a
peripheral, recover from a fault, switch context, …) [1][2].

### The vector table

The **vector table** is a table in memory that holds the reset value of the main
stack pointer, and the start addresses — called **exception vectors** — of every
exception handler [2]. Entry *N* holds the address of the handler for exception
number *N*: when exception *N* is taken, the core reads that entry and branches
to it. The table holds addresses only — no enable, priority, pending/active, or
fault-status bits — that's the registers' job, described above in "The role
registers play for interrupts/exceptions." The diagram below shows where the
table sits in the full memory map, and what each entry holds:

![Zoom-in memory map showing where the Cortex-M7 vector table lives: the full 4 GB map with the boot memory (default, at reset) and SRAM (optional VTOR-relocated copy) highlighted, next to the vector table's own offset layout — Initial SP, Reset, NMI, the fault handlers, the system-service handlers, and the external IRQn entries.](images/vector_table_memory_map.png)

For comparison, the vector table figure as published in ST's PM0253 [2] — the same
offset/exception-number/IRQ-number layout, straight from the primary source:

![Vector table figure reproduced from STMicroelectronics PM0253: columns for exception number, IRQ number, byte offset, and vector name, from offset 0x0000 (Initial SP value) through Reset, NMI, the fault handlers, SVCall, PendSV, SysTick, and IRQ0 upward to the highest implemented IRQ.](images/vector_table_STMprocessor_CortexM.png)

### Relocating the vector table

The bridge between the vector table and the registers is one register: **`VTOR`**
(Vector Table Offset Register, in the SCB) holds the *base address* of the vector
table, telling the core where the table currently lives [1].

After reset, the vector table is read from address `0x0000_0000` — aliased to
whichever boot memory the device selects at power-up [1]. Software can move it
elsewhere at runtime by writing the new base address into `VTOR`. This is routine
in a bootloader-to-application handoff — each image ships its own complete vector
table, and the bootloader must repoint `VTOR` at the application's table before
jumping to it (see [3.3 Bootloader](3.3-Embedded-Fundamentals-Boothloader.md) for
a worked example) — and in systems that copy the table into RAM to patch
individual handlers at runtime.

Relocation has to be done carefully:

- **Alignment.** `VTOR`'s base address must be aligned to the size of the table,
  rounded up to a power of two; the low alignment bits aren't writable, so an
  unaligned value is silently truncated and the core ends up reading a table
  that doesn't start where you think it does [1].
- **Populate before pointing.** The new table must be fully written — a valid
  stack-pointer entry and every handler address the image needs — *before*
  `VTOR` is updated. If an exception is taken in the gap between a half-written
  table and the `VTOR` write, the core vectors through garbage.
- **Mask interrupts during the switch.** Disable interrupts (e.g. raise
  `BASEPRI`, or clear the relevant `SHCSR` enables) while copying the table and
  writing `VTOR`, so nothing can fire mid-update and read a torn entry.
- **Serialize the write.** The table may sit behind a cache or a buffered memory
  type; a `DSB` followed by `ISB` after the copy and after the `VTOR` write
  ensures the core doesn't fetch a stale entry or keep running from the old
  table [1].
- **The first fetch is unaffected.** The boot-time Initial SP value and Reset
  vector are always read from the boot memory alias at `0x0000_0000`, before any
  software — including anything that touches `VTOR` — has run. Relocation only
  changes where *later* exceptions are vectored, not how the core comes out of
  reset [1].

### Exception stack frame

When the core takes an exception it automatically pushes eight 32-bit words onto
the currently active stack (MSP, or PSP if the exception preempted thread-mode
code running on the process stack) — this is called **stacking**, and the pushed
block is the **exception stack frame** [1]:

| Offset (from SP after stacking) | Register | Contents                                                            |
| ------------------------------- | -------- | ------------------------------------------------------------------- |
| `0x00`                          | `R0`     | General-purpose                                                     |
| `0x04`                          | `R1`     | General-purpose                                                     |
| `0x08`                          | `R2`     | General-purpose                                                     |
| `0x0C`                          | `R3`     | General-purpose                                                     |
| `0x10`                          | `R12`    | General-purpose (intra-procedure scratch)                           |
| `0x14`                          | `LR`     | Return address into whatever code was interrupted                   |
| `0x18`                          | `PC`     | Program counter at the point of interruption                        |
| `0x1C`                          | `xPSR`   | Program status (condition flags, execution state, exception number) |

Stacking these eight registers automatically — rather than leaving it to the
handler's prologue — is what lets any C function be used directly as an
exception handler: the core has already saved everything the calling convention
doesn't guarantee the callee will preserve, so the handler just runs and returns
[1]. (On an FPU-enabled Cortex-M7 build using the floating-point context, the
core can additionally stack `S0–S15` and `FPSCR` — *lazy* by default, meaning
space is reserved but the registers are only actually written if the handler
touches the FPU [1].)

### Exception entry

1. The core (via the NVIC) picks the highest-priority exception that is both
   pending and enabled, and that has higher priority than whatever is currently
   executing. It first finishes the instruction in progress — for a multi-cycle
   instruction (e.g. an integer divide, or a multi-register load/store) the core
   either lets it complete or abandons and restarts it after the exception,
   depending on the instruction [1][3].
2. It pushes the exception stack frame described above onto the active stack.
3. It reads the vector table entry for that exception number and loads it into
   the program counter.
4. It loads `LR` with a special **`EXC_RETURN`** value instead of a normal
   return address — a reserved bit pattern (`0xFFFFFFE1`/`E9`/`ED`, …) that
   encodes which stack and mode to return to, rather than an address to branch
   to [1].
5. It updates `IPSR` (the exception-number field of `xPSR`) to the exception
   number just taken, and begins executing the handler in **Handler mode**.

Because entry is symmetric with the priority scheme, a higher-priority exception
can itself be taken while a handler is running — the core simply repeats this
sequence, stacking a second frame on top of the first. This is what "nested" in
Nested Vectored Interrupt Controller means [1].

### Exception return

An exception handler doesn't return like an ordinary function. Instead, the
handler ends by loading the program counter with the `EXC_RETURN` value that was
placed in `LR` at entry — typically via `BX LR`, or implicitly via `POP {..., PC}`
if `LR` was itself stacked and restored. The core recognizes that value (its top
bits are not a valid code address) as a request to **unstack**, not a branch [1]:

1. It reads `EXC_RETURN` to determine which stack (MSP or PSP) and which mode
   (Thread or Handler) to return to, and whether an FPU frame needs unstacking.
2. It pops the eight-word exception stack frame back off that stack, restoring
   `R0–R3`, `R12`, `LR`, `PC`, and `xPSR` to their pre-exception values.
3. Execution resumes at the restored `PC`, in the restored mode — either back in
   the code that was interrupted, or in a lower-priority handler that was itself
   preempted.

If another exception is already pending at a high enough priority when the
current handler finishes, the core can skip re-stacking/unstacking entirely and
"tail-chain" straight into the next handler — an optimization that keeps
back-to-back interrupts cheap [1].

tail-chaining shortcut on the return side:

### Exception handling scenarios

The entry/return sequences above are the general case; the core takes shortcuts
whenever it can prove they're safe. Four scenarios cover how a given exception
actually gets serviced [1][3].

#### Normal (stacking) exception

Nothing else is in flight: the core runs the full entry sequence (stack,
vector-fetch, `IPSR` update), and unstacks on return. This is the baseline path
laid out in the activity diagram above.

#### Nested / pre-empted exception

A higher-priority exception is taken while a lower-priority handler is still
**active**. The core stacks a *second* frame on top of the first, and unstacks
back into the outer handler — still active the whole time, just suspended —
resuming exactly where it left off once the inner handler returns [1][3]:

![Pre-emption timing diagram: three interrupt request lines (IRQ1 highest priority, IRQ2, IRQ3) plotted against time above a Core Execution bar below. ISR2 runs in the foreground, is pre-empted mid-execution by higher-priority IRQ1's ISR1, which is itself pre-empted by IRQ3's ISR3; execution then unwinds back down through ISR1 to ISR2 resuming, and finally back to foreground code.](images/pre-emption.png)

#### Tail-chained exception

The current handler finishes and a *different* exception is already pending at
a high enough priority. Rather than unstack the finished handler's frame and
immediately stack a fresh one for the next exception, the core reuses the frame
already on the stack and jumps straight to the next handler's vector — turning
a stack+unstack pair into a single cheap vector fetch [1][3]:

![Tail-chaining timing diagram: IRQ1 fires first (Push, then ISR1 runs while IRQ1 is active); IRQ2 fires while ISR1 is still running. When ISR1 finishes, the core performs a "TC" (tail-chain) step directly into ISR2 — instead of a full pop/push pair — and only pops once ISR2 finishes and control returns to the foreground.](images/interrupts-tail-chaining-speedup.png)

#### Late-arriving exception

A higher-priority exception becomes pending *while* the core is still stacking
for a lower-priority one — after stacking has started but before the handler
begins executing. The core finishes the in-flight stacking (it's already
committed), then vectors to the higher-priority handler instead of the original
target, at no extra stacking cost. The displaced exception, still pending,
typically runs next via tail-chaining once the higher-priority handler returns
[1][3]:

![Interrupt late-arrival timing diagram: IRQ2 fires and stacking ("Push") begins for it, but before ISR2 starts, higher-priority IRQ1 fires during that same push. The core redirects into ISR1 first, tail-chains ("TC") into ISR2 once ISR1 finishes, then pops back to the foreground.](images/interrupt-late-arrival-speedup.png)

Tail-chaining and late arrival exist purely to cut latency: both avoid an
unstack/restack pair that would otherwise happen back-to-back for no functional
reason. Nesting, by contrast, is unavoidable whenever a strictly higher-priority
exception must run before a lower-priority handler can finish — see
[Exception state](#exception-state) in Concept Introduction for how the NVIC
tracks which of these is happening at any given moment.

## 5. Q&A

1. **What's the difference between PC and LR at exception entry vs. exception return?**

   A: At exception entry, the processor writes an `EXC_RETURN` value to `LR` — this indicates which stack pointer corresponds to the stack frame and what operation mode the processor was in before the entry occurred. At exception return, the processor is in Handler mode and loads that `EXC_RETURN` value into `PC`. The exception mechanism relies on this value to detect when the processor has completed an exception handler.

2. **Is the vector table the same thing as the exception stack frame?**

   A: No. The vector table is a fixed array of handler *addresses*, one per exception number, sitting wherever `VTOR` points — the core reads it once per exception to find where to jump. The exception stack frame is the transient block of *register values* (R0–R3, R12, LR, PC, xPSR, …) pushed onto MSP or PSP for that same exception, and popped again on return. One says where to go; the other preserves what to resume on exception return.

3. **Is VTOR the vector table?**

   A: No — `VTOR` is a register that holds the vector table's *base address*. The table itself is data sitting in Flash or SRAM at that address; `VTOR` just points to it, and relocating a vector table means writing a new address into `VTOR`, not moving the register.

4. **Is "the stack" the same as "the exception stack frame"?**

   A: No. The stack (main or process) is the whole memory region that keeps growing and shrinking with every push/pop over the program's entire life. An exception stack frame is one specific 8-or-more-word slice pushed onto whichever stack was active at the moment of a *single* exception — no different in kind from any function's local stack frame, just pushed by hardware instead of a compiler-generated prologue.

5. **How do IPSR, EPSR, APSR, and xPSR relate to each other?**

   A: They're not four separate registers — APSR (condition flags), IPSR (active exception number), and EPSR (Thumb state, IT/ICI bits) are three non-overlapping bitfields of the *same* 32-bit physical register, collectively called xPSR (or PSR). `MRS`/`MSR` can read or write them individually or in combination.

6. **What's the difference between the NVIC and the SCB?**

   A: Both are register blocks in the System Control Space, but they manage different things. The NVIC is the register block for the interrupts themselves — per-IRQ enable, pending, active, and priority. The SCB is the register block for the system-level exceptions and configuration around them — fault enable/status registers, `VTOR`, `AIRCR`, and the priority registers for the fixed system exceptions (SVCall, PendSV, SysTick, the fault handlers).
